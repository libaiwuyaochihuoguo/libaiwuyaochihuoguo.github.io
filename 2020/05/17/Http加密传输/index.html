HTTP加密方式：

  • 对称加密：

    1. 过程：(密钥加密，密钥解密)
      • 浏览器/服务器端用密钥加密明文—发送密文—-传输—-接收密文—服务器/浏览器端用密钥解密密文
    2. 优点：简单
    3. 缺点：
      • 方式一：接收方和发送方不需要传输密钥，比如浏览器预存所有网站的密钥，该方式不现实。
      • 方式二：接收方和发送方通过传输密钥得到密钥，该方式传输过程中易被中间人攻击。

  • 非对称加密：(私钥如何得到，如何做到公钥加密只有私钥能解？)

    1. 简介：服务器有两把密钥，一把叫公钥、一把叫私钥，浏览器只有一把接收到的公钥。用公钥加密的内容只有私钥能解开，私钥加密的内容只有公钥能解开。
    2. 过程：(私钥发，公钥解；公钥发，私钥解)
      • 服务器端以明文方式发送公钥—浏览器端接收公钥
      • 服务器用私钥加密明文—发送密文—-传输—-接收密文—浏览器用公钥解密密文(不可靠)
      • 浏览器用公钥加密明文—发送密文—-传输—-接收密文—服务器用私钥解密密文(可靠)
    3. 优点：第三个过程貌似可靠，因为只有服务器有私钥，能解开浏览器用公钥加密过的明文。
    4. 缺点：第二个过程不可靠，因为公钥可能被中间人攻击，中间人也能用劫获的公钥解开服务器用私钥加密过的明文。且非对称加密算法效率低。

  • 改良的非对称加密：

    1. 简介：服务器和浏览器端各拥有一把唯一的私钥(P1、P2)和接受到的公钥(C2、C1)。
    2. 过程：(公钥发，私钥解)
      • 服务器端以明文方式发送公钥C1—浏览器端接收公钥C1
      • 浏览器端以明文方式发送公钥C2—服务器端接收公钥C2
      • 服务器用公钥C2加密明文—发送密文—-传输—-接收密文—浏览器用私钥P2解密密文
      • 浏览器用公钥C1加密明文—发送密文—-传输—-接收密文—服务器用私钥P1解密密文
    3. 优点：貌似可靠
    4. 缺点：非对称加密算法效率低。

  • 非对称加密+对称加密：

    1. 简介：服务器有两把密钥，一把公钥、一把私钥，浏览器只有一把接收到的公钥。
    2. 过程：(公钥发密钥，私钥解密钥，密钥加密，密钥解密)
      • 服务器端以明文方式发送公钥C—浏览器端接收公钥C(不可靠)
      • 浏览器端随机生成一个用于对称加密的密钥S
      • 浏览器用公钥C加密密钥S—发送密文—-传输—-接收密文—服务器用私钥解密得到密钥S
      • 双方用密钥S进行对称加密传输
    3. 优点：貌似可靠且效率高
    4. 缺点：浏览器无法保证接收到的公钥是服务器发的公钥，比如中间人拥有公钥C‘、私钥P’，服务器拥有公钥C、私钥P
      • 中间人在过程一中劫取公钥C明文，将公钥C替换成自己的公钥C’，将公钥C‘发给浏览器。
      • 浏览器生成一个用于对称加密的密钥S。
      • 浏览器用公钥C’加密S，发送该密文，中间人劫取密文，用P‘解密得到密钥S。
      • 中间人用公钥C重新加密密钥S，发送给服务器，服务器用私钥P解密得到密钥S。
      • 三者都拥有密钥S，且浏览器和服务器端毫不知情。

  • 数字签名技术：

    1. 简介：数字证书相当于网站的”身份证”，而数字签名就是为了验证数字证书是否被篡改。
    2. 作用：用来保证上述方案中浏览器接收到的公钥是服务器发的公钥。
